## Что за продукт
Внутренняя система для небольшой мастерской (мыло, свечи, косметика ручной работы):
- инвентаризация сыпучего и жидкого сырья (раз в месяц, возможно чаще)
- учёт расходов и пополнений сырья
- учёт производства партий по рецептам со списанием сырья
- учёт продаж готовых изделий
- КПД сотрудников по сменам
- дашборд с аналитикой
- Telegram-бот для быстрых запросов (низкий приоритет, см. "Роадмап реализации" п.9)

Требование к простоте интерфейса: должно быть управляемо человеком без технического бэкграунда ("любая обезьяна должна справиться"). Рядовой сотрудник не видит и не знает о существовании базы данных/API — только страницы React-приложения.

## Технический стек
- **Backend:** FastAPI (`backend/app/`), роутер на раздел (`app/routers/{auth,ingredients,production,packaging,recipes,products,sales,counterparties,dashboard}.py`)
- **База данных:** PostgreSQL через SQLAlchemy 2.0 (`app/models.py`), миграции — Alembic (`backend/alembic/`), DSN — `postgresql+psycopg://` в `app/config.py`/`DATABASE_URL`
- **Frontend:** React 19 + TypeScript + Vite (`frontend/src/`), Tailwind для стилей; страницы в `pages/`, модалки/детейл-панели в `components/`, `lib/api.ts` (`apiFetch`) и `lib/auth.tsx` (`useAuth`) — общий слой запросов и сессии
- **Пароли:** хэшируются через `bcrypt`, plaintext нигде не хранится (`app/security.py`)
- **Авторизация:** JWT (`pyjwt`) — сервер без состояния, фронт хранит токен и шлёт его в `Authorization` header; роль проверяется на каждом роутере через `require_roles(...)` (`app/security.py`), а не только скрытием пункта меню
- **Роли/константы:** `app/constants.py` (`FOUNDER`, `DEVELOPER`, типы транзакций, `PRODUCT_REQUIRED_FIELDS`)
- **Бот:** Telegram-бот (шаг 9 роадмапа, ещё не реализован) — при реализации будет ходить в тот же Postgres/FastAPI, а не напрямую в БД

### Почему не Google Sheets
Исходно планировался Streamlit+gspread (Google Sheets как база) — простота и отсутствие инфраструктуры перевешивали. От этого отказались в пользу Postgres+FastAPI+React ради нормальной типизации данных, ролевой модели через JWT и SPA-интерфейса вместо Streamlit rerun-модели. Раздел "Структура данных" ниже описывает те же сущности, что раньше были листами Google Sheets — теперь это таблицы Postgres (см. `app/models.py`), но набор полей и связи между ними не менялись.

## Структура данных

Ниже — те же сущности, что раньше были листами двух Google Sheets-файлов (см. "Почему не Google
Sheets"). Теперь это таблицы одного Postgres, определённые в `backend/app/models.py`; разделения на
"файл с рабочими данными" и "закрытый файл с доступами" на уровне БД больше нет — это было решение
специфичное для ACL Google Sheets (права выдаются на файл целиком). Разграничение по ролям теперь
на уровне API (`require_roles`), не на уровне БД/файла.

**Company** — тенант (мультитенантность, добавлена 2026-07-16, см. отдельный раздел ниже)
`id, название, дата создания`
Почти все остальные таблицы ниже несут `company_id` — исключение только RecipeItem (скоуп
транзитивно через recipe_id/material_id).

**Materials** — справочник сырья
`id, название, категория (сыпучее/жидкое/тара), ед.измерения, мин.остаток, текущий остаток`
Категория «тара» добавлена при переносе реальных рецептов из «oinarri рецептуры»: упаковка (флаконы, крышки, этикетки) списывается по рецепту наравне с сырьём.

**Transactions** — единый журнал движений сырья (сердце системы)
`id, дата, material_id, тип (приход/расход/корректировка), кол-во, цена, recipe_id (если списание по производству), комментарий`
Остаток = сумма приходов − расходов ± корректировки, считается на лету SQL-запросом (не хранится статично).
`POST /api/production` перед списанием проверяет остаток каждого материала рецепта — если сырья не
хватает на запрошенное кол-во партий, возвращает 400 и не пишет ни одной транзакции (см. `production.py`).

**Recipes** — рецепты
`id, название, категория, что производим, выход партии, технология`
Поле «технология» добавлено при переносе реальных рецептов — пошаговая инструкция приготовления, у founder в исходнике была для каждого рецепта. Поле «категория» — выбор из уже существующих категорий Product (не свободный ввод, не отдельный справочник), состав рецепта обязателен при создании.

**RecipeItems** — состав рецепта
`recipe_id, material_id, кол-во на 1 партию`
В Sheets-эпоху здесь дублировались "название рецепта"/"название материала" (Founder читала лист
глазами). В Postgres это лишний денормализованный столбец, который может разъехаться при
переименовании — таблица хранит только id, названия при чтении джойнятся из Recipes/Materials.

**Products** — готовые изделия (мыло, свечи и т.д.), карточка товара
`id, название, категория, GTIN, состав, recipe_id, ТН ВЭД, декларация соответствия, срок действия РД`
Обязательные при создании: название, категория, GTIN (`app.constants.PRODUCT_REQUIRED_FIELDS`).
Остальное опционально. "Название рецепта" в БД не хранится — джойнится через `recipe_id` при чтении
(см. `app/routers/products.py`).

**Counterparties** — контрагенты-покупатели (юрлица)
`id, название, ИНН, КПП, ОГРН, юр.адрес, телефон, контактное лицо, комментарий`
Заполняются Founder/Developer в разделе «Контрагенты» и привязываются опционально к Sale при отгрузке.
Автоподстановка по ИНН (2026-07-19) — `GET /api/counterparties/lookup?inn=` ходит в
suggestions.dadata.ru (`app/config.py::DADATA_API_KEY`, пусто по умолчанию → 501, фронт
тихо остаётся в обычном ручном режиме, ничего не ломает). На успехе фронт
(`NewCounterpartyModal.tsx`) блокирует найденные поля (название/КПП/ОГРН/адрес) от
ручного редактирования — просьба Founder исключить случайные опечатки при заведении
контрагента; «Ввести вручную» снимает блокировку, если DaData ошиблась/устарела.

**Sales** — продажи готовых изделий
`id, дата, product_id, counterparty_id (опционально), кол-во, цена, комментарий`
`POST /api/sales` проверяет «готово к отгрузке» продукта (см. `products.py::_ready_to_ship_by_recipe`) —
если продано больше, чем произведено минус уже проданное, возвращает 400. Продукты без `recipe_id`
(не привязаны к рецепту) проверке не подлежат — для них «готово к отгрузке» не считается.

**ProductionLog** — журнал смен производства
`id, дата, worker_id, recipe_id, кол-во партий, время начала/окончания, кол-во брака, комментарий`
Источник данных для КПД. КПД = количество партий + скорость (время на партию) + брак/ошибки.
`GET /api/production/leaderboard` — отдельный «лидерборд»-эндпоинт (не dashboard/kpi): кол-во
произведённого (партии×выход_партии − брак) за сегодня и с начала месяца по каждому сотруднику,
доступен ВСЕМ ролям включая worker (осознанное решение Founder — мотивация, все видят, кто сколько
делает). Отдаёт только количество — цен/выручки в ProductionLog нет вообще, изоляция не на фронте,
а в том, что этих полей физически нет в ответе. Не путать с `dashboard/kpi` (founder/developer,
полная история помесячно) — тот остаётся закрытым, там нет такого ограничения по данным.

**PackagingLog** — журнал упаковки (раздел «Упаковка»)
`id, дата, worker_id, product_id, кол-во, брак, комментарий`
Фиксирует момент упаковки готового изделия после производства — отдельный шаг от ProductionLog
(производство сырой партии) и Sale (отгрузка). Пока не влияет на расчёт «готово к отгрузке» у
Product (тот считается от ProductionLog, см. `app/routers/products.py::_ready_to_ship_by_recipe`) —
это самостоятельный журнал учёта, увязка с «готово к отгрузке» не решена, см. "Отложенные темы".

**Feedback** — обратная связь для разработчика
`id, дата, автор, роль автора, сообщение, статус (новое/просмотрено/решено)`
Доступна для отправки всем ролям. Список и статусы видят только Founder и Developer.

**Users**
`id, ФИО, логин, хэш пароля (bcrypt), роль (founder/worker/developer), статус (активен/уволен), дата создания, телефон, мессенджер, адрес проживания, документ (текстовое поле, например паспорт/ИНН)`
Учётные записи создаёт только Founder (или Developer) через интерфейс приложения. Раньше это был
физически отдельный Google Sheets-файл (права выдаются на файл целиком) — в Postgres это обычная
таблица в той же БД, изоляция обеспечивается на уровне API (`require_roles`), не на уровне БД.
Телефон валидируется на бэке (`app/schemas.py`) — только цифры и `+ - ( )`, буквы отклоняются.

**LoginLog** — журнал входов (кто и когда авторизовался)
`id, user_id, дата и время`
Пишется на каждый успешный `POST /api/auth/login`. Список (`GET /api/auth/log`) виден только
Founder/Developer — выводится в разделе «Сотрудники».

| Свой КПД | ✅ | ✅ | ✅ |

| КПД всех сотрудников (полная история, dashboard/kpi), продажи, финансы, дашборд | ✅ | ❌ | ✅ |
| Управление рецептами | ✅ | ❌ | ✅ |
| Добавить/уволить сотрудника | ✅ | ❌ | ✅ |
| Техническая панель (логи, статус API, сброс кэша) | ❌ | ❌ | ✅ |
| Заводить новые компании (тенанты) | ❌ | ❌ | ✅ |

Worker НЕ видит чужой полный КПД (dashboard/kpi — история, брак, финансы) и вообще не знает о

production/leaderboard (см. ProductionLog выше): осознанно открытая всем ролям витрина только
количества произведённого за сегодня/месяц, отдельная от закрытого dashboard/kpi.

## Архитектурные принципы, которые нельзя нарушать
1. Ни один прямой доступ/ссылка на БД не должен быть виден в интерфейсе Worker.
2. Проверка роли — на каждом роутере программно (`require_roles(...)` в `app/security.py` как FastAPI-зависимость), а не только скрытием пункта меню на фронте — иначе можно обойти через прямой запрос к API.
3. Пароли — только bcrypt-хэш, никогда plaintext.
4. JWT — секрет только из `JWT_SECRET` (env), никогда не хардкодить. `get_current_user` перепроверяет статус пользователя в БД на каждый запрос (уволенный теряет доступ сразу, не дожидаясь истечения токена) — роль, `company_id` и `company_name` при этом берутся из payload токена, не перечитываются из БД, так что смена роли/компании/названия компании применяется только после переиздания токена (следующий логин), это осознанный компромисс, не баг (пользователь и так не переезжает между компаниями на практике). `company_id`/`company_name` — оба обязательны, отсутствие любого из двух форсирует чистый 401 (не просто одного `company_id`) — иначе фронтовый фоллбек на вайт-лейбле мог показать чужой бренд компании, у которой токен не нёс company_name (см. "Мультитенантность").
5. Мультитенантность: каждый роутер обязан фильтровать ВСЕ запросы по `user["company_id"]` и проставлять его при создании записей — утечка данных между компаниями через забытый фильтр является критическим багом, не обычным. RecipeItem — единственное исключение без своего company_id (скоуп транзитивно через recipe_id/material_id, оба проверяются на принадлежность компании при создании/изменении состава).

## Мультитенантность (добавлена 2026-07-16)

Проект перешёл от однотенантной мастерской к изоляции нескольких независимых компаний в одной
БД — Founder решил попробовать пустить сюда не только свою мыловарню, но и, например, 3Д-печатную
мастерскую (вариант А из обсуждения — реальная изоляция, не просто общность модели данных).

- **Company** — тенант. `company_id` есть почти на каждой таблице (см. "Структура данных").
- **Онбординг новой компании — два пути.** (1) `POST /api/companies` (`app/routers/companies.py`,
  2026-07-18) — Developer заводит компанию прямо из приложения (пункт «Компании» в меню профиля,
  только у Developer, Founder такого пункта не видит вообще). Создаёт компанию + сразу первого
  **Developer**-аккаунта этой компании — НЕ Founder, того заводит тот же разработчик отдельно после
  (через `create_founder.py --company-id`, п.2 ниже, или это станет отдельной кнопкой позже). Это
  единственный роутер, намеренно не фильтрующий по `user["company_id"]` — вся его задача cross-tenant
  (список всех компаний + создание новой), см. "Архитектурные принципы" п.5. (2) CLI-скрипт
  `backend/scripts/create_founder.py` остаётся для бутстрапа первой компании на чистой БД (там ещё
  некому зайти в интерфейс) и для добавления второго Founder к уже существующей компании:
  `python -m scripts.create_founder --company-name "Новая мастерская" --fio "..." --login ... --password ...`
  (или `--company-id <id>` вместо `--company-name`, не плодя дубликат компании).
  Оба пути — **не** самостоятельная регистрация: требуют существующего Developer-аккаунта с доступом
  к приложению/БД, не публичная форма (та отдельно и по-прежнему не в реализации, см. ниже).
- **Логин глобально уникален** (не per-company) — экран входа не спрашивает "какая компания", она
  определяется автоматически по аккаунту. `company_id` живёт в JWT-payload наравне с ролью (см.
  "Архитектурные принципы" п.4) — не перечитывается из БД на каждый запрос.
- **Раскладка дашборда** (`DashboardWidgetLayout`) — per-user с 2026-07-20 (была общая на всю
  систему до мультитенантности, затем per-company; см. п.7 роадмапа ниже) — уникальность
  `(company_id, user_id, widget_key)`.
- **Изоляция проверяется явно** — `backend/tests/test_multitenancy.py`, отдельный файл, не
  разбросано по остальным тестам. Кросс-компанийный доступ по чужому id всегда 404 (не 403) —
  не подтверждаем даже факт существования чужой записи.
- Первая компания (существующая мастерская, "oinarri") — результат бэкфилл-миграции
  `a772deab8f2a_multi_tenant_company_id.py`, фиксированный id `a0000001`.
- **Публичная self-serve регистрация** (2026-07-18, реализована) — `POST /api/auth/register`
  (`app/routers/auth.py`, без `require_roles`/`get_current_user` — единственный такой POST-эндпоинт
  в приложении), `RegisterPage.tsx` на `/register`. Создаёт компанию + первого **Founder**-аккаунта
  (не Developer — тот только для internal-онбординга, `POST /api/companies` выше) и сразу логинит
  (тот же `{access_token, user}`, что и `/login`). Переиспользует `attach_or_create_membership` —
  если введённый логин уже существует у другого человека, привязывает его Founder'ом новой компании
  (тот же "вариант А", тот же пароль-чек). Обязательные поля: название компании, ФИО, телефон, логин,
  пароль — телефон с прицелом на будущую SMS/Telegram-верификацию, сам механизм которой пока не выбран
  (см. ниже). Rate-limit 5/час по IP (`app/rate_limit.py`, in-memory, читает `X-Forwarded-For` — Render
  проксирует, `request.client.host` был бы IP прокси, не посетителя, security-review 2026-07-18 поймал
  до пуша). Капчи пока нет — осознанная задолженность (см. ниже). Публичный анонимный эндпоинт ловит
  400 от `attach_or_create_membership` и подменяет сообщение на общее — "неверный пароль" и "уволен"
  по отдельности превратили бы эндпоинт в оракул для перебора чужих логинов (тот же security-review).
  `RegisterCompanyRequest` (`schemas.py`) — `max_length` на company_name/fio/login (совпадает с длиной
  колонок Postgres — без этого анонимный запрос ронял бы 500 вместо чистого 422, SQLite в тестах эту
  ошибку не ловит, маскировала баг) и байтовая (не символьная) проверка пароля ≤72 байт — bcrypt≥4.2
  кидает `ValueError` на пароле длиннее, кириллица 2 байта/символ. Тесты — `test_register.py`, включая
  отдельный regression на изоляцию данных нового тенанта (не полагается только на общий механизм в
  `test_multitenancy.py`).
  **Задолженность (сознательно, не блокирует первый заход):** капча (hCaptcha/Turnstile — оба
  бесплатные), телефон собирается, но не верифицируется — механизм (SMS-код или через Telegram-бота,
  требует своего бота + Telegram-аккаунт у клиента) ещё не выбран Founder'ом.

## Мульти-компанийные пользователи (добавлена 2026-07-18)

Founder попросил: один и тот же человек (сначала — сам разработчик как Developer, но
потенциально и Founder/Worker) может обслуживать/работать сразу в нескольких компаниях,
без необходимости заводить отдельный логин на каждую.

- **CompanyMembership** (`app/models.py`) — связь user↔company↔role, `unique(user_id, company_id)`.
  `User` больше НЕ несёт `company_id`/`role` напрямую (были там до 2026-07-18, перенесены сюда
  миграцией `2963dcbe68c7_company_memberships.py`) — `User` теперь чистая личность (ФИО, логин,
  пароль, `status`, телефон/мессенджер/адрес/документ). **Все эти поля остаются глобальными на
  User** (уволен = уволен из всех компаний сразу; смена телефона/ФИО/документа в одной компании
  видна во всех, где состоит человек) — осознанное решение: это один реальный человек с одним
  именем/паспортом/телефоном в жизни, не разные личности по компаниям. Per-membership копии этих
  полей не запрашивались и не сделаны.
- **Вход с одной компанией** (99% пользователей) — не меняется ни на бит: `POST /api/auth/login`
  сразу отдаёт `access_token` + `user`, как и раньше.
- **Вход с несколькими компаниями** — `authenticate()` (`app/security.py`) не логинит сразу,
  отдаёт `{needs_company_choice: true, pending_token, companies: [...]}`. Фронт (`LoginPage.tsx`)
  показывает список компаний вместо навигации; `POST /api/auth/select-company` с `pending_token`
  + выбранным `company_id` финализирует вход. `pending_token` — отдельный короткоживущий (5 мин)
  JWT-тип (`PENDING_TOKEN_TYPE`), не полноценная сессия — годится только на этот один эндпоинт.
- **Переключение компании уже залогиненным** — `POST /api/auth/switch-company`, без повторного
  пароля (личность уже подтверждена), просто переиздаёт токен на другое членство того же
  человека. В сайдбаре (`Sidebar.tsx`) — блок «Другие компании» в меню профиля, виден только
  если у юзера больше одного членства.
- **JWT payload несёт `companies: [{id, name, role}]`** — список ВСЕХ компаний юзера, не только
  активной, посчитан на момент выдачи токена. Как и `role`/`company_name`, не перечитывается из
  БД на каждый запрос (см. п.4 архитектурных принципов) — смена состава компаний применяется
  только после переиздания токена (следующий логин/переключение), тот же осознанный компромисс.
  `get_current_user` фоллбечит на `[текущая компания]`, если поля нет (токены до 2026-07-18).
- **Существующий логин при создании — не ошибка, а приглашение** (вариант А, решение Founder),
  единый helper `attach_or_create_membership` (`app/security.py`) — используется в
  `POST /api/companies`, `POST /api/users` и `create_founder.py`, не три отдельные копии.
  Если введённый логин уже существует у другого человека:
  - **Требует правильный пароль существующего аккаунта.** Без этой проверки любой Founder/
    Developer мог бы привязать ЧУЖОЙ аккаунт к своей компании, зная только логин (не пароль),
    и через `reset-password` перехватить ГЛОБАЛЬНЫЙ пароль — это была реальная дыра, поймана
    security-review 2026-07-18 до пуша в прод, не гипотетическая. Пароль в форме поэтому НЕ
    игнорируется (в отличие от ФИО/телефона и т.п. — те действительно игнорируются, у
    человека уже есть свой аккаунт).
  - **Уволенный существующий аккаунт (status ≠ активен) — отклоняется**, не привязывается
    молча. Без этой проверки founder думал бы что "нанял" рабочего, а тот не смог бы зайти
    (status глобальный, увольнение из одной компании блокирует все).
  - Дубликат (тот же user_id уже состоит в ТОЙ ЖЕ компании) — по-прежнему ошибка 400.
  - Фронт (`NewStaffModal.tsx`/`NewCompanyModal.tsx`) показывает предупреждение, если бэк
    вернул `attached_existing: true` — иначе founder не узнал бы, что "новый сотрудник" на
    самом деле уже существующий человек, а не свежесозданный аккаунт.
- **Сброс пароля** (`POST /api/users/{id}/reset-password`) меняет пароль на `User`, то есть сразу
  во ВСЕХ компаниях, где состоит человек — это осознанное поведение (один пароль на личность),
  не баг.
- **`get_current_user` отклоняет `pending_token`** явно (проверка `payload.get("type")`) — без
  этой проверки pending-токен проходил `jwt.decode` как обычный и падал `KeyError`→500 вместо
  чистого 401 (был баг, пойман code-review 2026-07-18 до пуша).
- **`switch-company` тоже пишет `LoginLog`** (той компании, куда переключились) — иначе
  Founder/Developer этой компании не видел бы в истории входов доступ через переключалку,
  только через прямой логин.
- Тесты: `test_multi_company_login.py` (вход/select-company/switch-company, включая отказ
  чужого pending_token и обычного access_token вместо pending), `test_companies.py` и
  `test_users.py` (вариант А — привязка существующего логина, неверный пароль отклоняется,
  уволенный существующий аккаунт отклоняется).

  поток камеры при параллельной нагрузке (кто-то смотрит YouTube). Скоуп по-прежнему фазами:
  Phase 1 — только лайв без записи; Phase 2 — 3-дневный retention/перемотка/скорость после
  того, как связка роутера обкатана.
- **Публичная self-serve регистрация компаний** — реализована 2026-07-18, см. "Мультитенантность" →

## Обновление фронта после деплоя (добавлено 2026-07-18)
Вместо разлогина всех пользователей после каждого прод-деплоя (обсуждали, но это грубо —
задело бы и обычные рестарты, не только осознанные изменения) — баннер снизу «Доступно
обновление интерфейса», применяется строго по клику на кнопку. `GET /api/version`
(`app/config.py::APP_VERSION`, публичный, без авторизации) отдаёт `RENDER_GIT_COMMIT` —
Render сам прописывает эту переменную при деплое, ничего не надо менять в Dockerfile.
`UpdateBanner.tsx` (смонтирован в `App.tsx` вне `<Routes>`, виден на любой странице
включая `/login`) поллит раз в минуту, сравнивает с версией при первом успешном ответе.
Сессию/токен не трогает вообще — только подгружает новый JS/CSS через reload.

Правила на будущее:
- Новая функция с логикой → тест на неё.
- Починка бага → регрессионный тест.
- Обработка ошибки → тест, который эту ошибку триггерит.
- Условная ветка (if/else) → тест на оба пути.
- Не коммитить код, который ломает существующие тесты.
