## Что за продукт
Внутренняя система для небольшой мастерской (мыло, свечи, косметика ручной работы):
- инвентаризация сыпучего и жидкого сырья (раз в месяц, возможно чаще)
- учёт расходов и пополнений сырья
- учёт производства партий по рецептам со списанием сырья
- учёт продаж готовых изделий
- КПД сотрудников по сменам
- дашборд с аналитикой
- Telegram-бот для быстрых запросов

Требование к простоте интерфейса: должно быть управляемо человеком без технического бэкграунда ("любая обезьяна должна справиться"). Рядовой сотрудник никогда не видит и не знает о существовании Google Sheets — только форму в Streamlit.

## Технический стек
- **Frontend/приложение:** Streamlit, деплой на Streamlit Community Cloud
- **База данных:** Google Sheets через библиотеку `gspread` (доступ строго через сервисный аккаунт Google, ключ — в `st.secrets`, никогда не в коде и не виден пользователю)
- **Пароли:** хэшируются через `bcrypt`, plaintext нигде не хранится
- **Авторизация:** своя, на `st.session_state` (НЕ Google-логин сотрудника — у сотрудников нет гугл-аккаунта в системе, только логин/пароль внутри приложения)
- **Бот:** Telegram-бот, отдельный процесс (Railway/Render либо локально), общается с теми же Google Sheets через gspread

### Почему не SQLite и не Postgres
- SQLite отпадает: на Streamlit Community Cloud файловая система эфемерна, при переразвёртывании файл базы обнулится.
- Postgres/Supabase — избыточно для масштаба задачи и требует больше инфраструктуры/знаний, чем нужно на данном этапе. Пользователь осознанно выбрал простоту.
- Google Sheets — оптимальный компромисс: бесплатно, не исчезает, можно посмотреть глазами, легко чинить руками при необходимости.

## Структура данных

### Файл 1: "Мыловарня: Учёт" (рабочие данные, обычный доступ сервисного аккаунта)

**Materials** — справочник сырья
`id, название, категория (сыпучее/жидкое/тара), ед.измерения, мин.остаток, текущий остаток`
Категория «тара» добавлена при переносе реальных рецептов из «oinarri рецептуры»: упаковка (флаконы, крышки, этикетки) списывается по рецепту наравне с сырьём.

**Transactions** — единый журнал движений сырья (сердце системы)
`id, дата, material_id, тип (приход/расход/корректировка), кол-во, цена, recipe_id (если списание по производству), комментарий`
Остаток = сумма приходов − расходов ± корректировки, считается на лету через pandas (не хранится статично).

**Recipes** — рецепты
`id, название, что производим, выход партии, технология`
Поле «технология» добавлено при переносе реальных рецептов — пошаговая инструкция приготовления, у founder в исходнике была для каждого рецепта.

**RecipeItems** — состав рецепта
`recipe_id, название рецепта, material_id, название материала, кол-во на 1 партию`
Названия рядом с id — намеренное дублирование: Founder открывает этот лист глазами напрямую,
голые id ей не читаемы. id остаются главными для связи между таблицами (названия могут разъехаться
при переименовании рецепта/материала — id нет).

**Products** — готовые изделия (мыло, свечи и т.д.)

**Sales** — продажи готовых изделий
`id, дата, product_id, кол-во, цена, комментарий`

**ProductionLog** — журнал смен производства
`id, дата, worker_id, recipe_id, кол-во партий, время начала/окончания, кол-во брака, комментарий`
Источник данных для КПД. КПД = количество партий + скорость (время на партию) + брак/ошибки.

**Feedback** — обратная связь для разработчика
`id, дата, автор, роль автора, сообщение, статус (новое/просмотрено/решено)`
Доступна для отправки всем ролям. Список и статусы видят только Founder и Developer.

### Файл 2: "Мыловарня: Доступы" (ОТДЕЛЬНЫЙ закрытый файл — доступ имеет только сервисный аккаунт)

**Users**
`id, ФИО, логин, хэш пароля (bcrypt), роль (founder/worker/developer), статус (активен/уволен), дата создания, телефон, мессенджер, адрес проживания, документ (текстовое поле, например паспорт/ИНН)`

**Важно:** этот файл физически отдельный от рабочих данных, не просто отдельный лист — потому что права доступа в Google Sheets выдаются на файл целиком. Учётные записи создаёт только Founder (или Developer) через интерфейс приложения.

## Роли и права

| Действие | Founder | Worker | Developer |
|---|---|---|---|
| Внести производство/остатки | ✅ | ✅ | ✅ |
| Свой КПД | ✅ | ✅ | ✅ |
| КПД всех сотрудников, продажи, финансы, дашборд | ✅ | ❌ | ✅ |
| Управление рецептами | ✅ | ❌ | ✅ |
| Добавить/уволить сотрудника | ✅ | ❌ | ✅ |
| Техническая панель (логи, статус API, сброс кэша) | ❌ | ❌ | ✅ |

Worker НЕ видит чужой КПД (только свой) и вообще не знает о существовании Google Sheets — работает исключительно со страницей Streamlit.

## Архитектурные принципы, которые нельзя нарушать
1. Ни один прямой доступ/ссылка на Google Sheets не должен быть виден в интерфейсе Worker.
2. Проверка роли — на каждой странице программно (через `st.session_state`), а не только скрытием пункта меню — иначе можно обойти через URL/код.
3. Данные пользователей кэшировать (`st.cache_data`) с ручным сбросом кэша при добавлении сотрудника — не дёргать Sheets на каждый rerun.
4. Пароли — только bcrypt-хэш, никогда plaintext.
5. Учитывать особенности Streamlit rerun-модели при парсинге/фоновых операциях — избегать бесконечных перезапусков.

## Отложенные темы (пока не в реализации)
- **Интеграция с "Честным знаком"**: рассмотрена и отложена. Не все товары мастерской попадают под обязательную маркировку на данный момент. Если понадобится — это отдельный опциональный источник данных для Sales (например поле `chestny_znak_code`), не блокирует MVP.
- **Видеонаблюдение**: нужна вкладка для Founder и Developer. Технически не решено — зависит от типа камер (IP/RTSP в локальной сети vs облачный сервис камер) и от того, нужен ли живой просмотр или досточно архива. Если приложение остаётся на Streamlit Community Cloud (удалённый хостинг) — камеры должны быть доступны из интернета, локальные IP-камеры за роутером без доп. настройки не подойдут. Решение по этому пункту не финализировано.

## Роадмап реализации
1. Сервисный аккаунт Google + структура обоих файлов + seed-запись Founder
2. Каркас логина (bcrypt, `st.session_state`, разграничение ролей)
3. Раздел "Сырьё": инвентаризация, приход/расход
4. Раздел "Производство": смены, списание сырья по рецепту, брак → данные для КПД
5. Раздел "Продажи" + управление рецептами/продуктами
6. Управление сотрудниками (добавление/увольнение) — Founder/Developer
7. Дашборд: остатки, топ сырья по расходу, топ продаж, динамика трат, КПД
8. Техпанель для разработчика
9. Telegram-бот (команды: /остатки /траты_месяц /прогноз /топ_сырьё /топ_продажи)
10. Видеонаблюдение (после уточнения по камерам)
