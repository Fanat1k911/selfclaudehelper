## Что за продукт
Внутренняя система для небольшой мастерской (мыло, свечи, косметика ручной работы):
- инвентаризация сыпучего и жидкого сырья (раз в месяц, возможно чаще)
- учёт расходов и пополнений сырья
- учёт производства партий по рецептам со списанием сырья
- учёт продаж готовых изделий
- КПД сотрудников по сменам
- дашборд с аналитикой
- Telegram-бот для быстрых запросов

Требование к простоте интерфейса: должно быть управляемо человеком без технического бэкграунда ("любая обезьяна должна справиться"). Рядовой сотрудник не видит и не знает о существовании базы данных/API — только страницы React-приложения.

## Технический стек
- **Backend:** FastAPI (`backend/app/`), роутер на раздел (`app/routers/{auth,ingredients,production,packaging,recipes,products,sales,counterparties,dashboard}.py`)
- **База данных:** PostgreSQL через SQLAlchemy 2.0 (`app/models.py`), миграции — Alembic (`backend/alembic/`), DSN — `postgresql+psycopg://` в `app/config.py`/`DATABASE_URL`
- **Frontend:** React 19 + TypeScript + Vite (`frontend/src/`), Tailwind для стилей; страницы в `pages/`, модалки/детейл-панели в `components/`, `lib/api.ts` (`apiFetch`) и `lib/auth.tsx` (`useAuth`) — общий слой запросов и сессии
- **Пароли:** хэшируются через `bcrypt`, plaintext нигде не хранится (`app/security.py`)
- **Авторизация:** JWT (`pyjwt`) — сервер без состояния, фронт хранит токен и шлёт его в `Authorization` header; роль проверяется на каждом роутере через `require_roles(...)` (`app/security.py`), а не только скрытием пункта меню
- **Роли/константы:** `app/constants.py` (`FOUNDER`, `DEVELOPER`, типы транзакций, `PRODUCT_REQUIRED_FIELDS`)
- **Бот:** Telegram-бот (шаг 9 роадмапа, ещё не реализован) — при реализации будет ходить в тот же Postgres/FastAPI, а не напрямую в БД

### Почему не Google Sheets
Исходно планировался Streamlit+gspread (Google Sheets как база) — простота и отсутствие инфраструктуры перевешивали. От этого отказались в пользу Postgres+FastAPI+React ради нормальной типизации данных, ролевой модели через JWT и SPA-интерфейса вместо Streamlit rerun-модели. Раздел "Структура данных" ниже описывает те же сущности, что раньше были листами Google Sheets — теперь это таблицы Postgres (см. `app/models.py`), но набор полей и связи между ними не менялись.

## Структура данных

Ниже — те же сущности, что раньше были листами двух Google Sheets-файлов (см. "Почему не Google
Sheets"). Теперь это таблицы одного Postgres, определённые в `backend/app/models.py`; разделения на
"файл с рабочими данными" и "закрытый файл с доступами" на уровне БД больше нет — это было решение
специфичное для ACL Google Sheets (права выдаются на файл целиком). Разграничение по ролям теперь
на уровне API (`require_roles`), не на уровне БД/файла.

**Materials** — справочник сырья
`id, название, категория (сыпучее/жидкое/тара), ед.измерения, мин.остаток, текущий остаток`
Категория «тара» добавлена при переносе реальных рецептов из «oinarri рецептуры»: упаковка (флаконы, крышки, этикетки) списывается по рецепту наравне с сырьём.

**Transactions** — единый журнал движений сырья (сердце системы)
`id, дата, material_id, тип (приход/расход/корректировка), кол-во, цена, recipe_id (если списание по производству), комментарий`
Остаток = сумма приходов − расходов ± корректировки, считается на лету SQL-запросом (не хранится статично).
`POST /api/production` перед списанием проверяет остаток каждого материала рецепта — если сырья не
хватает на запрошенное кол-во партий, возвращает 400 и не пишет ни одной транзакции (см. `production.py`).

**Recipes** — рецепты
`id, название, категория, что производим, выход партии, технология`
Поле «технология» добавлено при переносе реальных рецептов — пошаговая инструкция приготовления, у founder в исходнике была для каждого рецепта. Поле «категория» — выбор из уже существующих категорий Product (не свободный ввод, не отдельный справочник), состав рецепта обязателен при создании.

**RecipeItems** — состав рецепта
`recipe_id, material_id, кол-во на 1 партию`
В Sheets-эпоху здесь дублировались "название рецепта"/"название материала" (Founder читала лист
глазами). В Postgres это лишний денормализованный столбец, который может разъехаться при
переименовании — таблица хранит только id, названия при чтении джойнятся из Recipes/Materials.

**Products** — готовые изделия (мыло, свечи и т.д.), карточка товара
`id, название, категория, GTIN, состав, recipe_id, ТН ВЭД, декларация соответствия, срок действия РД`
Обязательные при создании: название, категория, GTIN (`app.constants.PRODUCT_REQUIRED_FIELDS`).
Остальное опционально. "Название рецепта" в БД не хранится — джойнится через `recipe_id` при чтении
(см. `app/routers/products.py`).

**Counterparties** — контрагенты-покупатели (юрлица)
`id, название, ИНН, КПП, ОГРН, юр.адрес, телефон, контактное лицо, комментарий`
Заполняются Founder/Developer в разделе «Контрагенты» и привязываются опционально к Sale при отгрузке.

**Sales** — продажи готовых изделий
`id, дата, product_id, counterparty_id (опционально), кол-во, цена, комментарий`
`POST /api/sales` проверяет «готово к отгрузке» продукта (см. `products.py::_ready_to_ship_by_recipe`) —
если продано больше, чем произведено минус уже проданное, возвращает 400. Продукты без `recipe_id`
(не привязаны к рецепту) проверке не подлежат — для них «готово к отгрузке» не считается.

**ProductionLog** — журнал смен производства
`id, дата, worker_id, recipe_id, кол-во партий, время начала/окончания, кол-во брака, комментарий`
Источник данных для КПД. КПД = количество партий + скорость (время на партию) + брак/ошибки.

**PackagingLog** — журнал упаковки (раздел «Упаковка»)
`id, дата, worker_id, product_id, кол-во, брак, комментарий`
Фиксирует момент упаковки готового изделия после производства — отдельный шаг от ProductionLog
(производство сырой партии) и Sale (отгрузка). Пока не влияет на расчёт «готово к отгрузке» у
Product (тот считается от ProductionLog, см. `app/routers/products.py::_ready_to_ship_by_recipe`) —
это самостоятельный журнал учёта, увязка с «готово к отгрузке» не решена, см. "Отложенные темы".

**Feedback** — обратная связь для разработчика
`id, дата, автор, роль автора, сообщение, статус (новое/просмотрено/решено)`
Доступна для отправки всем ролям. Список и статусы видят только Founder и Developer.

**Users**
`id, ФИО, логин, хэш пароля (bcrypt), роль (founder/worker/developer), статус (активен/уволен), дата создания, телефон, мессенджер, адрес проживания, документ (текстовое поле, например паспорт/ИНН)`
Учётные записи создаёт только Founder (или Developer) через интерфейс приложения. Раньше это был
физически отдельный Google Sheets-файл (права выдаются на файл целиком) — в Postgres это обычная
таблица в той же БД, изоляция обеспечивается на уровне API (`require_roles`), не на уровне БД.
Телефон валидируется на бэке (`app/schemas.py`) — только цифры и `+ - ( )`, буквы отклоняются.

**LoginLog** — журнал входов (кто и когда авторизовался)
`id, user_id, дата и время`
Пишется на каждый успешный `POST /api/auth/login`. Список (`GET /api/auth/log`) виден только
Founder/Developer — выводится в разделе «Сотрудники».

## Роли и права

| Действие | Founder | Worker | Developer |
|---|---|---|---|
| Внести производство/упаковку/остатки | ✅ | ✅ | ✅ |
| Свой КПД | ✅ | ✅ | ✅ |
| КПД всех сотрудников, продажи, финансы, дашборд | ✅ | ❌ | ✅ |
| Управление рецептами | ✅ | ❌ | ✅ |
| Добавить/уволить сотрудника | ✅ | ❌ | ✅ |
| Техническая панель (логи, статус API, сброс кэша) | ❌ | ❌ | ✅ |

Worker НЕ видит чужой КПД (только свой) и вообще не знает о существовании базы данных/API — работает исключительно со страницами React-приложения.

## Архитектурные принципы, которые нельзя нарушать
1. Ни один прямой доступ/ссылка на БД не должен быть виден в интерфейсе Worker.
2. Проверка роли — на каждом роутере программно (`require_roles(...)` в `app/security.py` как FastAPI-зависимость), а не только скрытием пункта меню на фронте — иначе можно обойти через прямой запрос к API.
3. Пароли — только bcrypt-хэш, никогда plaintext.
4. JWT — секрет только из `JWT_SECRET` (env), никогда не хардкодить. `get_current_user` перепроверяет статус пользователя в БД на каждый запрос (уволенный теряет доступ сразу, не дожидаясь истечения токена) — роль при этом берётся из payload токена, не перечитывается из БД, так что смена роли применяется только после переиздания токена (следующий логин), это осознанный компромисс, не баг.

## Отложенные темы (пока не в реализации)
- **Упаковка ↔ «готово к отгрузке»**: PackagingLog — самостоятельный журнал, пока не участвует в расчёте
  «готово к отгрузке» у Product (тот по-прежнему = произведено по ProductionLog − продано). Не решено,
  должна ли отгрузка списываться из упакованного остатка вместо сырого произведённого — требует
  уточнения у Founder, есть ли расхождение между «произведено» и «упаковано» на практике.
- **Интеграция с "Честным знаком"**: рассмотрена и отложена. Не все товары мастерской попадают под обязательную маркировку на данный момент. Если понадобится — это отдельный опциональный источник данных для Sales (например поле `chestny_znak_code`), не блокирует MVP.
- **Видеонаблюдение**: нужна вкладка для Founder и Developer. Технически не решено — зависит от типа камер (IP/RTSP в локальной сети vs облачный сервис камер) и от того, нужен ли живой просмотр или достаточно архива. При удалённом хостинге backend/frontend камеры должны быть доступны из интернета, локальные IP-камеры за роутером без доп. настройки не подойдут. Решение по этому пункту не финализировано.

## Testing

**Статус:** `backend/tests/` — pytest-набор поверх FastAPI `TestClient` + SQLite in-memory
(`tests/conftest.py`: фикстуры `db_session`/`client`, `get_db` подменяется на сессию к SQLite,
токены минтятся напрямую через `create_access_token`, без прогона bcrypt-логина в каждом тесте
кроме `test_auth.py`). Реальный Postgres не трогается. `pytest`/`httpx` — в `backend/requirements-dev.txt`
(не в основном `requirements.txt`, чтобы не тянуть dev-зависимости на прод), установлены в `backend/.venv`.
Покрыто: auth (логин/уволенный/лог входов/ролевой доступ), ingredients (баланс приход/расход/корректировка),
production (проверка достаточности сырья — см. ниже, worker видит только свои записи), sales (проверка
«готово к отгрузке» — см. ниже), recipes (пустой состав → 422, ролевой доступ), packaging, counterparties
(валидация телефон/ИНН/КПП/ОГРН), users (создание/дубликат логина/увольнение отзывает доступ сразу),
dashboard (spend: только приходные транзакции с ценой, фильтр по датам; kpi: группировка по
месяцу+сотруднику; ролевой доступ).
Запуск: `cd backend && .venv/bin/python -m pytest tests/ -q`.

Правила на будущее:
- Новая функция с логикой → тест на неё.
- Починка бага → регрессионный тест.
- Обработка ошибки → тест, который эту ошибку триггерит.
- Условная ветка (if/else) → тест на оба пути.
- Не коммитить код, который ломает существующие тесты.
